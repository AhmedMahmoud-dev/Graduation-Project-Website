## New Service — AlertService

Create src/app/core/services/alert.service.ts

This service handles all alert API calls and localStorage caching.

localStorage keys used by this service:
- emotra_alerts_meta — stores the full paginated API response data array (items array only, not the wrapper)
- emotra_alerts_stats — stores the stats object (total_alerts, unread_alerts, critical_alerts, high_alerts)

All API calls use the JWT interceptor — never attach Authorization header manually.

Methods to implement:

getAlerts(page, pageSize, severity?, resolved?):
- Calls GET /api/alerts with query params
- Returns Observable<ApiResponse<AlertsPagedResult>>

getStats():
- Calls GET /api/alerts/stats
- Returns Observable<ApiResponse<AlertStats>>

resolveAlert(id):
- Calls PATCH /api/alerts/{id}/resolve
- Returns Observable<ApiResponse<AlertItem>>

deleteAlert(id):
- Calls DELETE /api/alerts/{id}
- Returns Observable<ApiResponse<boolean>>

### Data Loading Pattern (Stale-While-Revalidate)

On component init:
1. Read emotra_alerts_meta from localStorage → if exists, render list immediately
2. Read emotra_alerts_stats from localStorage → if exists, render stats immediately
3. Fire GET /api/alerts (page=1, pageSize=50) in background → on success update list + save to emotra_alerts_meta
4. Fire GET /api/alerts/stats in background → on success update stats + save to emotra_alerts_stats
5. Use takeUntilDestroyed on both subscriptions
6. On API failure: if cache exists keep showing cache silently, if no cache show inline error state — no toast on load failure

### Actions

Mark as Read (resolve):
- Show loading state on that specific button while request is in flight
- Call PATCH /api/alerts/{id}/resolve
- On success: update that item in the local list (resolved = true, resolved_at set), update emotra_alerts_meta, decrement unread_alerts in stats, update emotra_alerts_stats, show success toast
- On failure: show error toast, revert button state

Delete:
- Call DELETE /api/alerts/{id}
- On success: remove item from local list immediately, update emotra_alerts_meta, update stats counts accordingly, update emotra_alerts_stats, show success toast
- On failure: show error toast, item stays in list

View Session:
- Get client_id and analysis_v2_id from the alert item
- If client_id exists: check emotra_text_sessions then emotra_audio_sessions for matching client_id
- If found in localStorage: navigate to result immediately
- If not found: call GET /api/analysis/{analysis_v2_id}, save result to correct sessions array (with duplicate check by client_id), then navigate
- If API fails: show error toast

### Pagination
- Load first page (pageSize=50) on init
- If total_count > 50, show a "Load More" button at the bottom
- On Load More: fetch next page, append items to existing list
- Update emotra_alerts_meta with the full merged list after load more
